# Cargo 配置与技术栈规格

## 文件职责

`Cargo.toml` 是 Auteo 桌面应用的 Rust package 清单，统一声明应用名称、版本、Rust 语言版本以及直接依赖。它是编译、依赖解析和 macOS 打包版本号的唯一事实来源。

## 产品与工程目标

Auteo 是面向创作者的 AI 剪辑师产品。桌面端需要长期承载大体积媒体素材、时间线交互、GPU 渲染、音视频处理和本地 AI 任务，因此技术栈优先考虑原生性能、内存可控性、低延迟交互、业务核心可复用性和安装包体积，而不是优先追求 Web 技术迁移速度。

## Package 元数据

- package 名称使用小写 `auteo`，作为 Cargo crate 名和默认可执行文件名。
- 用户可见产品名称使用“Auteo”，由 macOS 应用包元数据和打包脚本负责转换。
- `Cargo.toml` 中的 `version` 同时作为 Rust package 版本和安装包产品版本，避免多处维护。
- 当前 edition 为 Rust 2024，以使用最新稳定语言语义并降低未来迁移成本。
- `rust-version` 固定为 `1.95`，与 `rust-toolchain.toml` 保持一致，使本地开发和构建环境可复现。

## Rust 选型原因

- Rust 可以在不依赖垃圾回收停顿的情况下提供内存安全，适合对延迟、内存占用和稳定性敏感的视频编辑工作负载。
- 音视频解码、帧处理、文件扫描和本地 AI 推理通常需要高 CPU 吞吐或原生库集成，Rust 能直接调用系统 API、C/C++ 库和 GPU 能力。
- UI、媒体处理和后台任务可以共享类型、数据模型和并发基础设施，减少跨语言边界带来的序列化与调试成本。
- Cargo 提供统一的依赖、测试和构建流程，便于后续拆分 workspace，同时不要求当前阶段提前搭建复杂的多 crate 架构。

## GPUI 选型原因

- GPUI 是 GPU 加速的原生 Rust UI 框架，渲染模型适合高刷新率、密集交互和自定义绘制较多的桌面编辑器。
- Auteo 计划中的时间线、波形、素材网格和画布需要精细控制布局、输入事件和绘制性能，GPUI 比 DOM/WebView 路径更接近底层渲染能力。
- GPUI 与业务层同为 Rust，可以复用类型和异步任务，不需要通过 JavaScript bridge 传递大批量媒体状态。
- Zed 已在真实桌面编辑器场景中持续使用 GPUI，为复杂原生工具类产品提供了有价值的工程参照。

GPUI 仍属于快速演进中的生态。其 API 稳定性、文档完整度、无头渲染能力以及 Windows/Linux 的生产可用性都需要持续验证，因此业务规则、媒体处理和 AI 能力不得与具体 GPUI 视图强耦合。

## gpui-component 选型原因

- `gpui-component` 提供按钮、主题、根视图和常用交互基础设施，减少从零实现控件状态和视觉一致性的成本。
- 组件库保持 GPUI 原生渲染路径，不引入 WebView，也不改变 Rust 单语言架构。
- 当前使用范围保持克制，只复用确有需要的基础组件；产品差异化较强的时间线和媒体画布应优先实现为独立业务组件。
- `gpui-component` 与 GPUI 版本存在兼容关系，升级时必须一起验证编译、交互和打包结果。

## 与 Electron 和 Tauri 的取舍

- Electron 的生态、招聘和跨平台一致性更成熟，但 Chromium 与 Node.js 会增加安装包体积和基础内存占用，大规模媒体状态还可能引入主进程、渲染进程之间的通信成本。
- Tauri 的安装包通常比 Electron 小，也便于复用 Web 前端生态，但核心界面仍依赖各平台 WebView；复杂编辑器在渲染一致性、自定义 GPU 绘制和大规模交互状态方面需要额外适配。
- Auteo 当前选择 GPUI，是因为产品更看重原生性能、Rust 端到端类型共享和编辑器级自定义绘制能力。若后续跨平台交付速度成为更高优先级，应基于真实原型和性能数据重新评估，而不是把当前选择视为不可变约束。

## 跨平台与无头架构约束

- GPUI 负责交互式桌面窗口，不应成为媒体处理、AI 编排或项目数据模型的唯一运行环境。
- 后续核心能力应设计为不依赖窗口生命周期的 Rust 模块，使 macOS、Windows、Linux GUI 和无头任务能够共享业务实现。
- 无头截图或离屏渲染不能默认等同于 GPUI 桌面窗口渲染；在 GPUI 提供稳定能力前，需要单独验证平台后端、字体、GPU 和截图一致性。
- 当前 package 只验证了 macOS ARM64 构建和安装。Windows、Linux 以及三端统一分发均属于待验证范围，不能在产品口径中宣称已经支持。

## 依赖与锁文件规则

- 直接依赖必须在 `Cargo.toml` 中显式声明，并避免引入功能重叠的第二套 UI 基础设施。
- `Cargo.lock` 必须提交到 Git。Auteo 是最终用户应用，锁文件用于固定完整依赖图，保证开发、CI 和发布构建尽可能一致。
- 新增依赖前应评估维护活跃度、许可证、二进制体积、编译时间、平台支持和安全风险。
- 升级 Rust、GPUI 或 `gpui-component` 时，必须同步更新本文件中的版本口径，并至少执行格式检查、`cargo check`、release 构建和目标平台打包验证。

## 已知限制与非目标

- 当前清单只包含 UI 启动所需依赖，尚未引入音视频编解码、项目持久化、AI 推理、网络服务或遥测能力。
- 当前阶段不提前建立大型 workspace、插件系统或通用渲染抽象；相关结构应在真实产品能力出现后按最小边界拆分。
- 技术选型说明记录当前产品阶段的工程依据，不代表对未来框架迁移或多平台适配作出永久承诺。
